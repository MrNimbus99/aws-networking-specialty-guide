Quiz 1 — Sample Question
Network Design (30%)
A gaming company uses Global Accelerator to route player traffic to game servers running on EC2 instances behind NLBs in us-east-1 and eu-west-1. Players in South America experience higher latency than expected. Network analysis shows traffic is routing to us-east-1 instead of the geographically closer endpoint. What is the MOST likely cause?
  • Global Accelerator doesn't support routing to South America
  • The eu-west-1 endpoint is marked unhealthy by Global Accelerator health checks
  • ✓ Traffic dial for the us-east-1 endpoint group is set to 100% and eu-west-1 endpoint group is set to 0%
  • CloudFront is required in addition to Global Accelerator for South American traffic
Show Explanation
✓ Why correctGlobal Accelerator uses traffic dials to control the percentage of traffic routed to each endpoint. If us-east-1 is set to 100% and eu-west-1 to 0%, all traffic routes to us-east-1 regardless of geography. This is a common misconfiguration during testing or gradual rollouts.
✗ Why A is wrongGlobal Accelerator supports routing from all global locations, including South America. It uses AWS edge locations worldwide to ingress traffic onto the AWS network.
✗ Why B is wrongIf eu-west-1 were unhealthy, traffic would correctly fail over to us-east-1. However, the question states this is unexpected behavior, suggesting a configuration issue rather than a health problem.
✗ Why D is wrongGlobal Accelerator operates independently and doesn't require CloudFront. They serve different purposes: Global Accelerator for TCP/UDP traffic optimization, CloudFront for HTTP/HTTPS content caching.
Quiz 2 — Sample Question
Network Design (30%)
An enterprise has multiple AWS accounts and VPCs. The security team requires all DNS queries from VPCs to be logged for compliance auditing. Which solution provides comprehensive DNS query logging?
  • ✓ Configure Route 53 Resolver query logging for all VPCs and send logs to a centralized S3 bucket
  • Enable VPC Flow Logs on all VPCs and filter for port 53 traffic
  • Configure VPC DNS server logs in CloudWatch
  • Enable CloudTrail logging for Route 53 API calls
Show Explanation
✓ Why correctRoute 53 Resolver query logging captures all DNS queries made from VPCs, including queries to private hosted zones, public domains, and forwarded queries. Logs can be sent to CloudWatch Logs, S3, or Kinesis Data Firehose for centralized compliance auditing.
✗ Why B is wrongVPC Flow Logs capture network traffic metadata (source/destination IP, port, protocol, bytes) but do not include DNS query details like the domain name being resolved. Filtering for port 53 shows that DNS traffic occurred, but not what was queried.
✗ Why C is wrongThere is no native "VPC DNS server logs" feature in CloudWatch. VPCs use Route 53 Resolver for DNS, and the correct logging mechanism is Resolver query logging, not CloudWatch-based DNS server logs.
✗ Why D is wrongCloudTrail logs Route 53 control plane API calls (e.g., CreateHostedZone, ChangeResourceRecordSets) but does not capture data plane DNS queries made by resources in VPCs. It audits management operations, not DNS resolution activity.

Quiz 6 — Sample Question
Network Design (30%)
A company is designing a multi-account network architecture in a single AWS Region. The design includes 15 VPCs across five AWS accounts. Every VPC must be able to communicate with every other VPC, and all inter-VPC traffic must pass through a central inspection VPC running third-party firewall appliances. Which connectivity approach meets these requirements with the LEAST operational overhead?
  • ✓ Deploy a Transit Gateway, attach all VPCs, and use Transit Gateway route tables to direct inter-VPC traffic through the inspection VPC
  • Create full-mesh VPC peering between all 15 VPCs and configure route table entries to direct traffic through the inspection VPC
  • Deploy AWS PrivateLink endpoint services in the inspection VPC for each application service
  • Use VPC sharing via AWS RAM to place all workloads in a single shared VPC and use network ACLs for traffic inspection
Show Explanation
✓ Why correctTransit Gateway acts as a centralized routing hub that supports transitive routing. By attaching all 15 VPCs and configuring Transit Gateway route tables, inter-VPC traffic can be routed through the inspection VPC where firewall appliances perform traffic inspection. This requires only 16 attachments (15 VPCs + 1 inspection VPC) and centralized route table management.
✗ Why B is wrongFull-mesh VPC peering for 15 VPCs requires 105 peering connections (n(n-1)/2). VPC peering is non-transitive, so routing traffic through a central inspection VPC via peering connections is operationally complex and doesn't scale. Each VPC would need route table entries pointing to the inspection VPC.
✗ Why C is wrongPrivateLink is designed for exposing specific services, not for general inter-VPC connectivity with centralized inspection. It would require creating separate endpoint services for every application and doesn't support transparent traffic inspection between VPCs.
✗ Why D is wrongVPC sharing allows multiple accounts to share subnets, but placing all workloads in a single VPC removes network-level isolation between workloads. Network ACLs are stateless and limited to 20 rules per ACL, making them insufficient for full traffic inspection compared to a dedicated firewall appliance.

Quiz 10 — Sample Question
Network Implementation (26%)
A company has enabled DNSSEC signing on a Route 53 public hosted zone and added the DS record to the parent domain. Users report that some resolvers are not validating DNSSEC for the domain. What should the network engineer verify FIRST?
  • Query the domain using dig +dnssec and verify that RRSIG records are returned in the response
  • Verify that DNSKEY records exist in the hosted zone by querying the zone directly
  • ✓ Check that DS records exist in the parent domain to confirm the chain of trust is established
  • Enable Route 53 Resolver query logging and check for DNSSEC validation failures
Show Explanation
✓ Why correctThe DS (Delegation Signer) record in the parent domain (.com, .net, etc.) is what establishes the chain of trust between the parent zone and the child zone's DNSKEY. If the DS record is missing or incorrect in the parent domain, validating resolvers cannot build the chain of trust and DNSSEC validation will fail. Verifying the DS record in the parent domain is the first troubleshooting step because it is the most common point of failure when DNSSEC signing is enabled but validation is not working.
✗ Why A is wrongWhile verifying RRSIG records confirms that Route 53 is signing responses, the presence of RRSIG records alone does not guarantee that validating resolvers can verify the signatures. If the chain of trust is broken at the parent level (missing or incorrect DS record), RRSIG records will exist but validation will still fail.
✗ Why B is wrongDNSKEY records are automatically managed by Route 53 when DNSSEC signing is enabled. They are unlikely to be the cause of validation failures. The more common issue is a missing or misconfigured DS record in the parent domain.
✗ Why D is wrongResolver query logging captures DNS queries made within VPCs but does not directly show DNSSEC validation status for external resolvers. The issue described affects external resolvers, which are outside the scope of Resolver query logging.

Quiz 12 — Sample Question
Network Implementation (26%)
A company has configured a Direct Connect connection with a private VIF. BGP is established but EC2 instances cannot communicate with on-premises. What should be verified for troubleshooting? (Select THREE)
  • ✓ Security groups on EC2 instances allow traffic from on-premises CIDR
  • Direct Connect bandwidth is sufficient
  • VPC Flow Logs are enabled
  • ✓ VPC route tables have routes to on-premises CIDR pointing to Virtual Private Gateway
  • ✓ BGP is advertising routes correctly from both sides
  • Jumbo frames are enabled
Show Explanation
✓ Why correctWhen BGP is established but EC2 instances cannot reach on-premises, the issue lies in the data-plane path: VPC route tables must have entries pointing on-premises CIDRs to the Virtual Private Gateway, security groups on the EC2 instances must permit traffic from on-premises IP ranges, and BGP must be advertising the correct prefixes in both directions so that return traffic is routed properly.
✓ Why A (Security groups) is correctSecurity groups are stateful firewalls on each ENI. If the on-premises CIDR is not allowed inbound, packets arriving via the VGW are dropped at the instance level even though the network path exists.
✓ Why D (VPC route tables) is correctThe VPC route table associated with the EC2 instance subnets must contain a route for the on-premises CIDR with the Virtual Private Gateway as the target. Without this route, return traffic from EC2 has no path back to on-premises.
✓ Why E (BGP route advertising) is correctBGP session being 'established' only means the peers exchanged OPEN/KEEPALIVE messages. If the customer gateway is not advertising on-premises prefixes to AWS (or AWS is not advertising VPC CIDRs back), the data plane has no forwarding entries and traffic is black-holed.
✗ Why B (Direct Connect bandwidth) is wrongThe question states BGP is up, so the physical link is operational. Insufficient bandwidth causes performance degradation or packet loss under load, not a complete inability to communicate.
✗ Why C (VPC Flow Logs) is wrongFlow Logs are a monitoring and troubleshooting tool that records traffic metadata. Enabling them does not affect connectivity in any way.
✗ Why F (Jumbo frames) is wrongJumbo frames (MTU 9001) are an optional performance optimization on Direct Connect. Standard 1500-byte MTU works by default, so jumbo frame settings do not prevent basic connectivity.

The monthly Resolver endpoint charges are significant. The network team wants to reduce costs while maintaining DNS resolution for all VPCs and on-premises. What is the MOST cost-effective approach?
  • Use public DNS servers for all resolution
  • Create Resolver endpoints in each VPC
  • ✓ Create Resolver endpoints in one shared services VPC, route DNS traffic via Transit Gateway, and share Resolver rules across accounts via AWS RAM
  • Forward all DNS queries to on-premises DNS servers

You would still need at least one set of Resolver inbound endpoints, and all DNS traffic from AWS to on-premises would need to traverse the hybrid connection, adding latency.

Network Security, Compliance, and Governance (24%)
A financial services company hosts a public-facing web application behind an Application Load Balancer (ALB) in a public subnet. The security team has observed SQL injection attempts, cross-site scripting probes, and high-volume requests from specific geographic regions. They need a solution that inspects inbound HTTP traffic and blocks these threats before they reach the application. Which approach BEST addresses these requirements?
  • ✓ Associate AWS WAF with the ALB and configure rate-based rules, geo-match conditions, and managed rule groups for OWASP threats
  • Add inbound deny rules to the subnet Network ACLs for known malicious IP ranges
  • Restrict security group inbound rules to port 443 only
  • Enable AWS Shield Standard on the ALB
Show Explanation
✓ Why correctAWS WAF associated with the ALB allows the team to create rate-based rules, geo-match conditions, and SQL injection / XSS match statements that inspect HTTP requests before they reach the application. Because the ALB is the internet-facing entry point, placing WAF here gives centralized Layer 7 protection for every target group behind it. WAF also integrates with AWS Managed Rules, reducing the effort to maintain signatures.
✗ Why B is wrongNetwork ACLs operate at Layer 3/4 (IP and port). They cannot inspect HTTP headers, query strings, or request bodies, so they cannot detect SQL injection, XSS, or perform application-layer rate limiting.
✗ Why C is wrongSecurity groups are stateful Layer 3/4 filters. Like NACLs, they have no visibility into HTTP request content and cannot block application-layer attacks.
✗ Why D is wrongAWS Shield Standard is automatically included at no extra cost and protects against volumetric/network-layer DDoS, but it does not inspect HTTP request content. Shield Advanced adds DDoS response team support and cost protection but still relies on WAF rules for application-layer filtering.

Quiz 19 — Sample Question
Network Security, Compliance, and Governance (24%)
A financial services company requires Layer 2 encryption on all AWS Direct Connect dedicated connections between its data center and AWS. The security team needs to ensure that traffic is encrypted at the Ethernet frame level before it enters the AWS backbone. Which approach meets this requirement?
  • ✓ Enable MACsec (IEEE 802.1AE) on 10 Gbps or 100 Gbps dedicated Direct Connect connections using a MACsec-capable router at the customer end
  • Create an IPsec VPN tunnel over a Direct Connect public virtual interface to encrypt all traffic at Layer 3
  • Enable MACsec on a hosted Direct Connect connection provided by an AWS Direct Connect Partner
  • Configure TLS 1.3 on the Direct Connect connection using an AWS-managed certificate from ACM
Show Explanation
✓ Why correctMACsec (IEEE 802.1AE) provides Layer 2 hop-by-hop encryption on Direct Connect dedicated connections. It is supported on 10 Gbps and 100 Gbps dedicated connections. The customer must use a MACsec-capable router and configure a Connection Key Name (CKN) and Connectivity Association Key (CAK) pair. This encrypts Ethernet frames before they traverse the Direct Connect link.
✗ Why B is wrongIPsec over a Direct Connect public VIF provides Layer 3 encryption, not Layer 2 Ethernet-frame-level encryption. While IPsec is a valid encryption option for Direct Connect traffic, it does not meet the specific requirement of Layer 2 encryption.
✗ Why C is wrongMACsec is only supported on dedicated Direct Connect connections, not on hosted connections provided by AWS Direct Connect Partners. Hosted connections share the partner's physical link, so MACsec cannot be configured by the end customer.
✗ Why D is wrongTLS operates at Layer 4/5 (transport/session layer), not Layer 2. Direct Connect connections do not support TLS configuration at the link level. TLS is used for application-layer encryption (e.g., HTTPS), not for encrypting the physical link.

The company wants to reduce latency for Asian users while keeping infrastructure costs low and avoiding multi-Region application deployments. Which solution meets these requirements?
  • Deploy a full application stack in an Asia Pacific Region and use Route 53 latency-based routing to direct users to the nearest Region
  • ✓ Configure Amazon CloudFront with the origin in us-east-1, enable Origin Shield in an Asia Pacific Region, and use a cache policy with a high TTL for video segments
  • Use AWS Global Accelerator with a single endpoint group in us-east-1 to leverage the AWS global network for Asian users

Show Explanation

✗ Why A is wrongDeploying a full application stack in an Asia Pacific Region would reduce latency but directly contradicts the requirement to avoid multi-Region application deployments and keep costs low. Running duplicate infrastructure is expensive.
